## A3. Vital sign flowsheet

### A3-Q1 `a3-stream-sample-identity` — Stream-sample identity, correction, and fulfillment

- **Question:** `vitals.jsonl` rows have no event id. Three coupled sub-problems lack a clean substrate expression today: (a) addressing a single bad sample for correction, (b) refuting a monitor window with a manual recheck, (c) fulfilling an `intent.monitoring_plan` cadence with a stream window rather than an event-layer `action.measurement`.
- **Context:** Sample-keylessness is a deliberate scale choice (per A3 §2 claim (b)) — order-of-10⁴ samples per shift on continuous telemetry would crush events.ndjson if every sample carried an id. But the trade-off bites at three points: `links.corrects` and `links.supersedes` cannot target stream rows; ADR 003 routes fulfillment through an `action` that `supports` an observation, and a stream observation has no id to cite; and a monitor-vs-RN reconciliation today exists only at the assessment-citation layer (ADR 010 roles) with no substrate-level statement of "this monitor cycle is disputed." A4 (MAR) will exercise the same pattern when post-administration vitals fulfill response-window monitoring intents.
- **Options:**
  1. **Deterministic `sample_key`** = `hash(subject, encounter_id, name, sampled_at, source.kind, source.ref?)` on each row, plus `observation.subtype = vital_correction` events that target the key via a payload field (`data.target_sample_key`). Solves (a) and supports (b) and (c) by giving the stream a stable addressable handle without event-izing every sample.
  2. **Promote disputed / fulfilled windows to events.** The RN (or fulfillment author) writes an `observation.vital_sign` whose `data.summarizes_window: "vitals://…"` explicitly stands in for the stream window; standard `links.corrects` and `action.measurement.supports` then apply. Generalizes to "summary-of-window" beyond this case but keeps stream rows id-less.
  3. **New link kinds with URI targets:** `links.refutes_window` / `links.fulfills_window` carrying a `vitals://` URI plus `basis`. Adds URI-as-link-target as a substrate concept; composes awkwardly with the existing event-id link walker.
  4. **Hybrid (current implicit state):** continuous-stream samples satisfy cadence-derived openLoops without explicit fulfillment; one-shot ordered measurements use `action.measurement`; correction is handled only at the assessment layer via ADR 010 roles. No new substrate primitives.
- **Researcher's lean:** Option 1 (sample_key) for the addressability primitive plus Option 2 (summarize_window) as the human-readable pattern when a disputed or fulfilled window deserves an event of its own; Option 4 as the default for run-of-the-mill continuous fulfillment. Avoid Option 3 — URI-as-link-target deserves its own ADR before being introduced. Do NOT promote all corrected vitals to events; that collapses the stream-vs-event distinction.

### A3-Q2 `a3-shared-metrics` — Shared metrics across A1 and A3 (ABG / lactate)

- **Question:** Arterial pH, PaO₂, PaCO₂, and lactate appear both as lab observations (A1, drawn ABG/VBG) and as vital-frame fields (A3, from the Pulse engine or derived estimates like SpO₂-inferred PaO₂). How should the substrate represent the shared metric?
- **Context:** Pulse emits `pao2_mmHg`, `paco2_mmHg`, `ph`, and `lactate_mmol_l` in every frame at sim-clock rate (see `~/pi-rn/pi-sim/scripts/types.ts:VitalFrame`). In real practice these values are produced only at ABG draw time (minutes to hours apart). The substrate needs a discipline that (a) does not imply false precision (a patient does not get a new PaO₂ every 2 seconds), (b) lets assessments cite the right evidence source when both exist, (c) avoids `trend(metric: "lactate")` silently unioning two semantically different streams.
- **Options:**
  1. **A1 canonical; A3 suppresses.** The ingest translator drops these fields from `vitals.jsonl` entirely. Only ABG-draw events exist for these metrics. Simplest; most truthful; discards Pulse-emitted numbers that are realistic but not clinically produced.
  2. **Both first-class; `source.kind` distinguishes.** A3 streams the Pulse values with `source.kind: monitor_extension`; A1 writes the ABG as `observation.lab_result` with `source.kind: lab_interface_hl7`. Assessments cite whichever supports the claim. Requires `trend()` to filter by source or union carefully; consumers must understand the distinction.
  3. **A3 derived-only with `links.supports` back to A1.** Vital-frame ABG-like fields write as `observation.vital_sign` events (not stream rows) carrying `links.supports: [<lab_result_id>]` and treated as display-convenience. Preserves pi-sim's capacity to drive agent reasoning on continuous "estimated" ABG for training while keeping clinical audit anchored on A1.
  4. **Per-profile filter.** Profile flag on the ingest translator suppresses or emits these fields. Defers the decision to scenario configuration.
- **Researcher's lean:** Option 1 is cleanest for the clinical substrate but loses pi-sim's capacity to drive agent reasoning on ABG derangements without a scheduled ABG. Option 2 is most truthful about the data-generation process but pushes work onto every trend consumer. **Option 3 is the best compromise**: preserves pi-sim's teaching value while keeping the canonical record aligned with real clinical practice. Recommend Option 3, with Option 1 as the v0.3 default if agent-facing trend complexity becomes a problem.

### A3-Q3 `a3-oxygen-context` — Oxygen-delivery context: requirement, source-of-truth, and `currentState` axis

- **Question:** SpO₂ interpretation depends on room air vs supplemental O₂, flow, FiO₂, device, and (in some patients) target range. Three coupled sub-questions: (a) Is inline `context.o2_device` mandatory on every SpO₂ sample, or derivable at query time from the active `observation.context_segment{segment_type: o2_delivery}` covering `sampled_at`? (b) When inline context and active context-segment (or active oxygen `intent`) disagree, which is canonical? (c) Does the active context-segment set get a dedicated `currentState(axis: "context")` view, fold into `axis: "all"`, or live as a separate `activeContext()` helper?
- **Context:** A SpO₂ of 91% on room air is alarming; on HFNC 40 L FiO₂ 0.5 it is alarming for different reasons. Today (per A3 §2 claim (c)) seed `patient_001` carries `context: {o2_device, o2_flow_lpm}` inline on every sample; ADR 005 introduces `observation.context_segment` as the canonical interval representation, but the seed has not been migrated. The substrate currently has no way to surface "current context" cleanly to a view consumer — `currentState.ts` has axis-specific dispatch for vitals / problems / constraints / intents but no `"context"` value. Migration discipline matters: if inline context is dropped abruptly, every prior chart breaks; if it is kept as the source of truth, ADR 005 becomes ornamental.
- **Options for (a):**
  1. **Mandatory inline.** V-VITAL-03 errors when SpO₂ lacks `context.o2_device`. Simple; preserves single-row sample truth; doubles down on the per-sample-context pattern that ADR 005 was meant to retire.
  2. **Derive at query time.** Inline `context.o2_device` becomes optional and read-only-as-fallback; the active `context_segment` is canonical. V-VITAL-03 errors when SpO₂ lacks *both* inline context AND a covering `context_segment`. Most aligned with ADR 005.
  3. **Inline allowed, validated against active state.** Inline context is permitted but V-VITAL-06 warns on disagreement with the active segment; canonical answer for any query is the active segment. Migration-friendly; preserves sample truth; catches drift.
- **Options for (b)** (when sources disagree): Active `context_segment` wins; the inline divergence becomes a `quality.flags` entry or a separate validator warning. If an active oxygen `intent` (e.g., "NC at 4 L") also exists and disagrees with the segment, the segment wins (it represents what was actually applied, not what was ordered).
- **Options for (c):**
  1. **Add `axis: "context"`.** Returns `{[segment_type]: EventEnvelope}` keyed by segment_type of the active interval as of `asOf`. Matches existing axis pattern; minimal new surface.
  2. **Fold into `axis: "all"`.** Extends the existing `"all"` return with `context: Record<string, EventEnvelope>`. No new axis value; one round-trip covers everything bedside UI needs.
  3. **Separate `activeContext()` helper.** Keep `currentState` axes focused on clinical-state axes; give context its own named view. More discoverable; effectively a seventh view primitive.
  4. **Do nothing.** Consumers walk `observation.context_segment` events manually. Worst ergonomics.
- **Researcher's lean:** **(a) Option 3** — inline allowed, validated against active state — gives the cleanest migration story and matches GPT's "preserves sample truth; validation catches drift" instinct. **(b) Active segment wins** — segments record what was applied, not ordered, which matches clinical reality. **(c) Option 1** — add `axis: "context"`, follows the established axis pattern, plays naturally with the existing `CurrentState` discriminated union, and avoids a seventh named view. Option 2 (fold into `"all"`) follows additively without blocking Option 1.

### A3-Q4 `a3-early-warning-score` — Early-warning score storage (NEWS2 / MEWS / qSOFA)

- **Question:** Should early-warning scores such as NEWS2, MEWS, and qSOFA be derived-only, or stored as `assessment.subtype = risk_score` events when reviewed/acted on?
- **Context:** Scores are computed from vital samples and a local scoring rule (NEWS2 uses RR, SpO₂ on / off oxygen, systolic BP, pulse, consciousness, temperature, with supplemental-oxygen adjustment). Storing every computed score as a canonical event creates stale derived truth that diverges as the underlying samples change; never storing reviewed scores weakens audit for escalations triggered by score thresholds (a NEWS2 of 7 should drive a documented response under most local protocols). The middle ground is "compute live; persist when clinically used."
- **Options:**
  1. **Derived-only via view layer.** Never store score claims as canonical events. Every consumer recomputes from samples + scoring rule. Audit of "did the score trigger an escalation" must reconstruct the score at the time of action.
  2. **Store only when a clinician/agent asserts or acts on a score** as `assessment.subtype = risk_score` carrying `data.algorithm` (e.g., `"news2"`), `data.algorithm_version`, `data.score`, `data.parameter_snapshot` (the values that produced it), and `data.reasoned_asOf`. `links.supports` cites the vitals windows used.
  3. **Store every computed score as event stream.** Every recompute writes an event. Simplest audit path; potentially massive event volume; risk of supersession churn as parameters update.
- **Researcher's lean:** **Option 2.** Continuous computed scores remain derived; clinically reviewed or acted-on scores become assessment claims. The `risk_score` shape already exists in CLAIM-TYPES; this question is mostly about establishing a discipline rule and documenting the payload contract. Option 3 is rejected on volume grounds; Option 1 is rejected because audit needs the snapshot-at-time-of-action and reconstruction is brittle when scoring rules version forward.
- **Adjacent rule:** When an `intent.monitoring_plan` declares a score threshold (e.g., `data.alert_policy: {news2_score >= 7}`), the absence of a `risk_score` assessment after the threshold is crossed should fire OL-VITAL-03 (sustained unresolved alarm) — i.e., scores can drive open loops even when no canonical alert event exists.

### A3-Q5 `a3-alarm-and-artifact-events` — Alarm and artifact event class

- **Question:** Two coupled sub-problems share the same architectural shape. (a) Is alarm state purely derived (computed per query from latest sample + active monitoring_plan thresholds), or do canonical `observation.alert` events fire on state transitions, on sustained crossing past policy window, or both? (b) When a monitor reading is `quality.state: invalid` and the artifact materially affected workflow (false alarm, false escalation, alarm-fatigue contributor), is the inline `quality.flags` enough or does a parallel `observation.subtype = device_artifact` event also fire? Both questions share the same trade-off: substrate scale vs auditability.
- **Context:** `pi-sim/vitals/current.json` and the monitor harness compute `alarms: [MAP_LOW, LACTATE_HIGH]` per frame as a simple threshold test. Emitting one alert event per frame would produce thousands of events per shift and defeat the "vitals are not events" discipline. NPSG.06.01.01 specifically cares about the sustained-alarm case (alarm fatigue, failure-to-respond). `validate.ts:isActiveAlertPlaceholder` already anticipates an alert shape at the `resolves` layer without it being finalized. OL-VITAL-03 depends on some concrete representation. Symmetrically: `quality.state: invalid` samples are filtered from `currentState` and trend, but when an artifact triggered a false alarm or wasted clinical attention, the chart should preserve evidence of *why* the value was discarded — not silently drop it.
- **Options for (a) — alert events:**
  1. **Emit on state transitions** — `alert_fire`, `alert_sustained_past_policy`, `alert_cleared`. Three event kinds per alarm episode. Durable; compact; lets `resolves` partners target the fire or sustained event by id. Bounded by physiological reality (alarms don't transition at sim-clock rate).
  2. **Emit only the durable sustained event** — only NPSG.06.01.01-relevant sustained alarms become canonical. Smallest surface; may under-audit transient but clinically meaningful alarms (short asystole runs, brief desats during suctioning).
  3. **Alert-state purely derived.** `openLoops(kind: vital_alert)` computes current state from latest sample + threshold + monitoring_plan.alert_policy; no canonical event written. Minimal substrate; loses historical auditability of alarm timing.
  4. **Hybrid.** Option 3 for real-time state + Option 2 for historical audit — sustained event written when alarm crosses policy window; real-time UI reads from derived state.
- **Options for (b) — invalid-sample handling:**
  1. **Inline `quality` only.** Keep all readings in `vitals.jsonl` with `quality.state` and `quality.flags`; views filter invalid samples; no separate artifact event. Traceable ingest preserved; no event spam.
  2. **Drop at ingest.** Invalid samples never reach `vitals.jsonl`; logged outside chart for engineering audit. Simplest; loses clinical context for alarm-fatigue analysis.
  3. **Hybrid.** Keep all readings inline (Option 1) plus emit `observation.subtype = device_artifact` events only when the artifact materially changed care or required audit (e.g., triggered a false escalation, drove a probe re-site). Composes cleanly with Option (a)-1 or (a)-4: an artifact can `supports` (or `resolves`) a transient alert event, closing the audit loop.
- **Researcher's lean:** **(a) Option 1** for alert events — closest to how real monitoring systems record alarm life-cycle, aligns cleanly with ADR 009 `resolves` semantics (the resolving event targets the `fire` or `sustained` event by id), and bounded by physiology so volume is manageable. **(b) Option 3** for invalid samples — keep `vitals.jsonl` traceable, emit `device_artifact` only when clinically consequential. Do not silently drop invalid data if it affected workflow. **Joint payoff:** `device_artifact` events can `resolves` short-lived `observation.alert` events caused by the artifact, producing a clean audit trail of "alarm fired → artifact identified → alarm cleared without intervention" that is currently impossible to express. Validator rule: when `(a)-1` is adopted, the three kinds (`fire`, `sustained`, `cleared`) must appear in ordered triples per episode, and `device_artifact.links.resolves` is permitted to close `alert_fire` events even before `sustained` is reached.
