# RALPLAN - Agent Canvas UI Iteration

Status: approved for implementation direction; source edits not made in this planning pass.

## Requirements Summary

The handoff matches the intended direction with one important refinement: artifact review should move out of the inline Agent Canvas detail and into a floating review window. Implementation should stay generator-native in `scripts/agent-canvas.ts`, regenerate `docs/prototypes/pi-chart-agent-canvas.html`, and preserve the chart-first cockpit frame.

Required UI direction:

- Keep Agent Canvas as a separate left-nav tab alongside Overview.
- Keep the patient banner and compact vitals strip above the main shell.
- Keep the cockpit left sidebar navigation.
- Make the bottom area the Pi-agent chat interface, with scrolling history and an adjustable/resizable height affordance.
- Make the right drawer the generated artifact scratchpad.
- Open artifact cards into a floating full-review window before stage, commit, or discard.
- Add a small upcoming task/checker/ticker in the scratchpad, including `q1h vitals at 1000` and `Zosyn due at 1200`.
- Demote the event stream to lower/secondary page placement so it does not dominate Agent Canvas.
- Tighten drawer alignment, spacing, and right-side visual weight.

## Evidence

- Existing intake snapshot `.omx/context/pi-chart-v0-2-agent-canvas-20260426T020508Z.md` records the same desired outcome: cockpit/Overview plus Agent Canvas, bottom chat, and generated artifact scratchpad.
- `package.json` exposes `agent-canvas`, `typecheck`, and `test` scripts.
- `scripts/agent-canvas.ts` already renders the top patient bar/vitals, left nav, tabbed Overview/Agent Canvas shell, right artifact drawer, chat panel, and generated artifact data.
- Current mismatch: artifact cards switch to an inline detail panel, while the handoff/reference screenshot asks for a floating full-artifact review window.
- Current partial match: chat history scrolls and the textarea resizes, but the chat area itself is not adjustable.

## RALPLAN-DR Summary

### Principles

1. Chart-first cockpit: patient context, vitals, and nav remain stable across Overview and Agent Canvas.
2. Agent boundary clarity: chat is process; artifacts are the reviewable and committable product.
3. Prototype-native scope: keep the work in the TypeScript generator and generated HTML; add no dependencies.
4. Review before write: artifact actions must happen from a full review surface.
5. Dense clinical ergonomics: remove blank weight while preserving scanability.

### Decision Drivers

1. The current generator already has the correct data-backed surfaces, so the work is a layout and interaction iteration.
2. The design reference and screenshot clearly favor bottom chat plus side/floating scratchpad review.
3. Verification is straightforward: regenerate, typecheck, test, screenshot.

### Viable Options

| Option | Approach | Pros | Cons | Verdict |
|---|---|---|---|---|
| A | Minimal CSS/content patch on the current three-column view. | Smallest diff. | Misses floating review and adjustable bottom chat. | Rejected. |
| B | Targeted generator iteration: preserve shell, move chat to an adjustable bottom dock, make right drawer scratchpad + ticker, open artifacts in floating review. | Matches handoff while preserving repo-native generation. | More CSS/JS state work. | Chosen. |
| C | Port the full React reference prototype. | Highest reference fidelity. | Larger rewrite, weaker data-backed generator path, unnecessary runtime complexity. | Rejected. |

## Acceptance Criteria

1. Agent Canvas remains a tab reachable from the left nav; Overview remains available.
2. Patient banner and vitals strip remain above the shell in both tabs.
3. Left sidebar navigation stays visible and aligned.
4. Event stream is lower/secondary, not the dominant Agent Canvas content.
5. Bottom Pi-agent chat has scrollable history and a visible adjustable/resizable height affordance.
6. Right drawer is artifact scratchpad with counts/list/cards plus an upcoming-task ticker.
7. Artifact open/select launches a floating review window with full content, metadata, provenance/confidence, and stage/commit/discard actions.
8. Stage/commit/discard update visible status in both scratchpad and review window.
9. No new dependency is introduced; output remains generated by `scripts/agent-canvas.ts`.

## Implementation Steps

1. Preserve the current `render()` topology: doc band, patient bar, vitals strip, shell, left nav, chart area, right drawer.
2. Refactor the Agent Canvas body so artifact detail is not permanently inline; use a lighter summary/work area and a floating review window controlled by existing artifact JSON.
3. Add a scratchpad task/checker render helper with prototype tasks: `q1h vitals at 1000` and `Zosyn due at 1200`.
4. Convert the Agent Canvas chat into a bottom adjustable dock with scrollable history, resize bounds, and stable content padding.
5. Keep or move the event stream to lower secondary placement, especially outside the primary Agent Canvas work area.
6. Tighten drawer CSS: aligned header/subhead, compact ticker, independently scrolling artifact list, reduced blank right-side weight.
7. Regenerate `docs/prototypes/pi-chart-agent-canvas.html`.

## Risks And Mitigations

- Floating review may obscure chart context: keep it bounded, closeable, and visually anchored while top banner/vitals stay visible.
- Bottom chat may overlap content: add stable bottom padding and min/max resize bounds.
- Static JS can drift from initial markup: keep artifact status rendering centralized.
- Screenshot verification may be environment-sensitive: run browser capture from the terminal where Playwright launch works.

## ADR

### Decision

Choose the targeted generator iteration: preserve the cockpit shell and implement bottom chat, scratchpad ticker, and floating artifact review in `scripts/agent-canvas.ts`.

### Drivers

- Existing generator already owns the correct data-backed prototype surface.
- User handoff and screenshot annotations specify layout/interaction corrections rather than a new app architecture.
- No new dependency is needed.

### Alternatives Considered

- Minimal patch: rejected because it would leave artifact review inline and chat insufficiently adjustable.
- Full React reference port: rejected because it expands scope and weakens the repo-native static generation path.

### Why Chosen

This path addresses the actual mismatch while keeping the implementation small, reversible, and compatible with the current prototype generator.

### Consequences

- `scripts/agent-canvas.ts` and regenerated `docs/prototypes/pi-chart-agent-canvas.html` will change.
- CSS/JS state grows modestly for dock resizing and floating review.
- Visual screenshot review becomes a required acceptance gate.

### Follow-Ups

- If this prototype becomes production UI, extract reusable view helpers or move the surface into a proper app layer.
- Keep source-of-truth language visible: chat is process; committed artifacts are chart writes.
