# Building pi-chart's live monitor on Pulse foundations

**The pi-chart monitor module should be a passive, browser-side bedside display that consumes `pi-sim/vitals/current.json` via a small Node SSE/WebSocket bridge, modeled visually on Pulse Explorer and architecturally on the Pulse Unity Asset's three-component pattern (`PulseEngineDriver` / `PulseDataLineRenderer` / `PulseDataNumberRenderer`).** All Kitware-published Pulse UIs (Explorer, Unity Asset, Unreal Plugin) are **Apache 2.0** at source, so layout, color, alarm taxonomy, and renderer patterns are legally borrowable with attribution; none of the actual code is portable to a TypeScript/browser stack. The dominant risk is not technical — it is UX: a live monitor that looks like the chart of record will be mistaken for one. Defend against that with explicit "Live Monitor" labeling, no commit affordances, and freshness/stale/offline indicators tied to a heartbeat on `current.json`. There is no first-party Kitware web/WASM Pulse build, so pi-chart's monitor is genuinely new ground rather than a port; that is good for clean architecture and bad for prior art.

## Pulse UI surfaces relevant to pi-chart

Three Kitware UIs are directly relevant, plus two minor references. **Pulse Explorer** is the canonical visual reference; the **Pulse Unity Asset** is the canonical *component* reference; the **Pulse Unreal Plugin** is a confirming third data point. Python/Jupyter plotting and an older ParaView prototype exist but are peripheral to a bedside module.

**Pulse Explorer** — `https://gitlab.kitware.com/physiology/explorer`, Apache 2.0, C++/Qt desktop application originally prototyped on ParaView, now a standalone Qt Widgets app distributed as prebuilt Windows/macOS/Linux binaries. v3.0 (March 2022, blog at `kitware.com/pulse-explorer-3-0-release/`) added a refined vitals-monitor layout, a mechanical-ventilator panel, and the `Arrhythmia` action UI. The vitals screen renders **numeric tiles for HR, ABP (sys/dia/MAP), SpO₂, EtCO₂, RR, core temperature**, and time-series strips for **ECG (lead III), arterial pressure (used as the pleth proxy), and a capnogram-like trace**. Hero image: `https://pulse.kitware.com/Images/PulseExplorer-Vitals.png`; full screen: `https://www.kitware.com/main/wp-content/uploads/2022/03/ExplorerScreen-1024x646.jpg`. Treat Explorer as **design reference only** — Qt/MOC/VTK code does not port to web, and its action dialogs/scenario editor mutate sim state and are explicitly out of scope for pi-chart.

**Pulse Unity Asset** — `https://gitlab.kitware.com/physiology/PulseUnityAsset/` (also reachable as `physiology/unity`), Apache 2.0 at source, Unity Asset Store EULA on the packaged distribution at `assetstore.unity.com/packages/tools/integration/pulse-physiology-engine-139773` (v4.3.1, Feb 6 2025). A 100%-C# mirror of an earlier release exists at `https://github.com/agirault/PulseUnity` and is browseable when GitLab's Anubis bot-stopper blocks fetches. The asset ships a `VitalsMonitor` Unity scene "based on the design of the Pulse Explorer vital sign monitor" (Kitware blog, Feb 28 2019: `kitware.com/pulse-is-now-available-on-the-unity-asset-store/`). Its three core components are the most directly portable design pattern in the entire Pulse ecosystem: **`PulseEngineDriver`** advances Pulse one tick per Unity frame (analogous to a browser `requestAnimationFrame` or server tick loop), **`PulseDataLineRenderer`** draws a sweep/scroll waveform on a canvas with configurable Y-min/Y-max, and **`PulseDataNumberRenderer`** renders a single numeric tile bound to a named Pulse channel. The monitor surfaces sim time, ECG, HR, ABP sys/dia, SpO₂, EtCO₂, RR, temperature, airway CO₂ partial pressure, and blood volume. **First files to inspect:** `Scripts/PulseEngineDriver.cs`, `Scripts/PulseDataLineRenderer.cs`, `Scripts/PulseDataNumberRenderer.cs`, `Scenes/VitalsMonitor.unity`. Logic patterns are adaptable to TypeScript with Apache-2.0 attribution; the native `Plugins/`, UGUI Canvas code, custom Editor inspectors, and `PulseActionOnClick` (which mutates sim state) must **not** be ported.

**Pulse Unreal Plugin** — `https://gitlab.kitware.com/physiology/unreal` and `unrealengine.com/marketplace/en-US/product/pulse-physiology-engine`, announced June 28 2022 (`kitware.com/kitware-and-lumeto-develop-pulse-unreal-plugin-for-medical-simulation-and-training-on-unreal-engine/`). Apache 2.0 at the GitLab source; the Marketplace copy adds Epic's Marketplace EULA on top, so **always pull from GitLab if borrowing**. C++ Actor Component plus Blueprint API plus an example UMG vitals monitor scene targeting UE 4.26/5.0. Confirms the same channel set as Explorer/Unity. Reusability is **architectural pattern only** — UMG widgets and Blueprint code don't translate to web.

**Peripheral references worth knowing about, but not building from:** the Pulse Jupyter notebooks at `gitlab.kitware.com/physiology/jupyter` ship a Python/matplotlib utility (added in Pulse 4.2 per `pulse.kitware.com/version.html`) that turns a Pulse CSV into a vitals-monitor-style plot — useful as a layout/column spec validator but not as a visual model. **"PulseDataRequester" and "PulseStudio" do not exist as distinct products** despite their natural-sounding names; the actual abstraction is `SEDataRequest` / `SEDataRequestManager` / `SEEngineTracker` inside the engine. There is **no Kitware-published Trame app, no Emscripten/WASM build, and no first-party JavaScript Pulse demo** — pi-chart's web monitor is not retreading prior Kitware work.

## Pulse data semantics pi-chart must preserve

The engine ticks at a **fixed 0.02 s = 50 Hz** internal step (`pulse.kitware.com/_system_methodology.html`) and emits one data-request row per tick when tracking is enabled. Scalar vitals like HR, RR, and the per-cycle ABP aggregates update at most once per cardiac/respiratory cycle but the column carries forward, so a downstream reader at any rate ≤ 50 Hz sees consistent values. The canonical bedside channel set comes from Pulse's own `_scenario_file.html` "Vitals Monitor" example and is what pi-sim should be writing into `current.json`:

| Pulse property (exact string) | Native unit string | Display unit | Display precision |
|---|---|---|---|
| `HeartRate` | `1/min` | bpm | integer |
| `SystolicArterialPressure` / `DiastolicArterialPressure` | `mmHg` | mmHg | integer |
| `MeanArterialPressure` | `mmHg` | mmHg | integer |
| `ArterialPressure` (instantaneous, waveform-capable) | `mmHg` | mmHg | n/a — plot only |
| `RespirationRate` | `1/min` | /min | integer |
| `PulseOximetry` (preferred SpO₂ — optical, includes CO-Hb) | `unitless` 0–1 | % (×100) | integer |
| `OxygenSaturation` (pure SaO₂) | `unitless` 0–1 | % (×100) | integer |
| `CoreTemperature` (preferred bedside temp) | `degC` | °C or °F | 1 decimal |
| `SkinTemperature` (used in stock scenario but lower than core) | `degC` | °C | 1 decimal |
| `EndTidalCarbonDioxidePressure` | `mmHg` | mmHg | integer |

**Critical naming subtleties:** Pulse's `OxygenSaturation` is pure SaO₂; `PulseOximetry` is the optical-equivalent value that includes carboxyhemoglobin and is what a real bedside SpO₂ probe reads — **pi-chart should display `PulseOximetry`** if pi-sim exposes it, falling back to `OxygenSaturation` only as a last resort. Saturations are **0–1 unitless**, not percent; multiply by 100 for display. Temperature should prefer `CoreTemperature` over the `SkinTemperature` used in the stock "Vitals Monitor" scenario, because Pulse's `Hyperthermia`/`Hypothermia` events fire on core, not skin.

**Simulation time** is exposed by the engine as `GetSimulationTime(unit)` returning real seconds elapsed since `InitializeEngine` (which returns at sim time = 0). State files like `bin/states/StandardMale@0s.json` confirm the convention. Display as `HH:MM:SS` from elapsed seconds; do **not** pretend it is wall-clock time. Show wall-clock separately if needed for ops, but the monitor's "clock" is sim time.

**Waveform availability is asymmetric and matters for pi-chart's roadmap.** Pulse natively exposes a 50 Hz **arterial-pressure waveform** via the `ArterialPressure` instantaneous channel, a **Lead-3 ECG** waveform via `{Category:"ECG", PropertyName:"Lead3ElectricPotential", Unit:"mV"}` interpolated from a digitized PhysioNet trace (Lead 1 and Lead 2 return NaN unless re-stabilized — confirmed by Kitware's Aaron Bray on Discourse threads `discourse.kitware.com/t/unity-ecg-data/1027` and `/electrocardiogram-output/1371`), and **respiratory flow/pressure** waveforms at 50 Hz. It does **not** expose a pleth waveform — Explorer and the Unity Asset both fake a pleth by plotting `ArterialPressure_mmHg` (Aaron Bray, Discourse, Feb 27 2024). It does **not** expose a continuous capnogram as a single property; you must request the gas-compartment partial pressure of CO₂ at the airway (`Category:"GasCompartment", CompartmentName:"Carina", SubstanceName:"CarbonDioxide", PropertyName:"PartialPressure", Unit:"mmHg"`) and sample at 50 Hz.

**Alarms come from the engine's event system, not from UI thresholds** — both Explorer and the Unity Asset rely on Pulse's `SEEventManager` enter/exit callbacks rather than reimplementing comparator logic. The full `eEvent` enumeration (per `pulse.kitware.com/_c_d_m_tables.html`) includes patient events (`Tachycardia`, `Bradycardia`, `Tachypnea`, `Bradypnea`, `Hypoxia`, `Hypercapnia`, `Hyperthermia`, `Hypothermia`, `CardiacArrest`, `HypovolemicShock`, `CardiogenicShock`, `IrreversibleState`, etc.), equipment events (oxygen-bottle exhaustion, relief valves), and seventeen ventilator alarms (`HighPressureAlarmTriggered`, `LowOxygenSaturationAlarmTriggered`, `ApneaTimeAlarmTriggered`, etc.). **Heart rhythm is separate** — `Asystole`, `CoarseVentricularFibrillation`, `PulselessVentricularTachycardia`, etc. live in the `eHeartRhythm` enum on the Cardiovascular system, not in `eEvent`. pi-chart should treat `pi-sim/vitals/current.json` as authoritative for which events/rhythms are active and surface a compact alarm row driven by that list rather than recomputing thresholds in JavaScript.

## Recommended pi-chart architecture

**Data path: file → tiny Node bridge → SSE → React monitor.** Have a small Node/Express service `chokidar`-watch `pi-sim/vitals/current.json` and push the parsed frame over **Server-Sent Events** to the browser whenever the file changes. SSE beats raw fetch polling because it provides automatic reconnection and a clean "EventSource closed" signal that maps directly to "OFFLINE"; it beats WebSocket because the data flow is one-directional (sim → display) and SSE has no negotiation overhead. The bridge should also emit a heartbeat event every 1 s carrying the file's `mtime` and parsed `simTime` even when the file hasn't changed, so the browser can compute freshness independently. If pi-sim writes more often than ~4 Hz, throttle on the server side to 4 Hz max for the SSE stream — the human eye doesn't benefit from faster numeric refresh, and waveforms (if present in `current.json`) should be batched as arrays inside one frame rather than spammed as individual updates.

**Strict separation from any future EHR adapter.** The display path must read `current.json` and write nothing. A separate, later adapter — running in its own process — will read the same file (or an equivalent stream) and append to `vitals.jsonl` for chart-of-record persistence. The monitor module must not import that adapter, must not call its writers, and must not be reachable from any UI affordance that implies "save to chart." Architecturally: one Node process serves the SSE bridge with read-only filesystem access; a separate process owns `vitals.jsonl`. This keeps the display path live even when persistence is broken, and prevents a UI that "feels live" from accidentally becoming the document of record.

**Avoiding false-truth UX.** Label the module **"Live Monitor"** in the header, not "Vitals" or "Chart." Never render a "Save" or "Commit to chart" button in this module. Show a persistent footer caption like *"Live snapshot from pi-sim · not part of the medical record"* in muted type. If the rest of pi-chart has a charting workflow, the monitor should be visually distinct (different background, monitor-style monospace numerics, dark theme à la real bedside displays) so users never confuse the snapshot with a documented value. When a clinician needs to chart a vital they observed on the monitor, the workflow should be a manual transcription into the (separate) charting form, not a one-click capture from the monitor — that constraint is what keeps documentation deliberate.

**Freshness, stale, and offline states.** Define three thresholds against the SSE heartbeat: **fresh** when the last frame is < 2 s old, **stale** when 2–10 s old (display values dimmed, a yellow "STALE" pill in the status row, last-update timestamp shown), **offline** when > 10 s old or the EventSource is in `CLOSED` state (replace numeric values with `---`, show a red "NO SIGNAL" banner, freeze any waveforms with a flat trace). A **fourth state, "invalid frame"**, fires when `current.json` parses but a required field is missing or NaN — render `---` for that specific tile and keep the rest live. The freshness logic must run client-side off `Date.now() - frame.receivedAt` so it remains correct even if the server clock drifts.

## Milestone-1 monitor feature set

Scope discipline matters more than feature breadth here, so the milestone splits cleanly into four bands.

**Must-have for M1:** numeric tiles for HR, BP (sys/dia with MAP shown smaller below), RR, SpO₂, and temperature; a sim-time clock in `HH:MM:SS`; a freshness/connection indicator with the four states above; an alarm row that renders any active Pulse events from `current.json` as colored chips (red for life-threatening, yellow for warnings, neutral for informational); and a "Live Monitor — not chart truth" footer caption. Tiles should use a high-contrast monospace font for numerics and clinical-monitor color conventions (HR green, SpO₂ cyan, ABP red, RR yellow, EtCO₂ white) sourced from Explorer screenshots.

**Should-have if time allows:** color-coded ranges that flip a tile's accent color when a value crosses a normal range (use the Pulse event presence as the source of truth for "abnormal" rather than hard-coded thresholds — that way you stay aligned with engine semantics); smoothed value transitions (CSS transition over 300 ms or a small exponential filter on the displayed integer); a subtle pulse animation on the alarm row when a new event fires; and a single basic waveform strip if `current.json` carries an `arterialPressure` array, rendered as a sweep trace on Canvas.

**Defer to M2+:** full multi-waveform fidelity (ECG + ABP + capnogram simultaneously); audible alarms (regulatory and accessibility considerations are non-trivial); multi-patient view; trends/sparklines beyond the live waveform; user-configurable thresholds; and any historical playback.

**Explicitly out of scope, forever, for this module:** any control that mutates sim state (start/stop/pause/scenario load — those belong in pi-sim's own UI if they exist); any EHR documentation or charting affordance; any patient identifier from the EHR side; any "Save snapshot to chart" button; and any code path that writes to `vitals.jsonl` or any persistence target.

## TypeScript/Node/browser implementation specifics

**Stack choice.** React with plain DOM for the tiles (each tile is a memoized component subscribed to one channel of a Zustand or Jotai store), Canvas 2D for any waveform strip (SVG works for static charts but not for 50 Hz sweep traces), and a custom `useEventSource` hook that owns the SSE connection plus a `useFreshness` hook that derives stale/offline state from the latest frame's `receivedAt`. Avoid React for the waveform render itself — drive it with `requestAnimationFrame` directly against an imperative ring buffer the SSE handler appends into. `setInterval` is fine for the 1 Hz freshness recompute but wrong for waveform draw.

**Update cadence.** Server emits at most 4 Hz over SSE for numerics, with waveform samples (if present) batched as arrays inside the same frame. Browser updates tile DOM at the SSE arrival rate. Browser updates waveform Canvas via `requestAnimationFrame`, consuming from the ring buffer. Engine internal 50 Hz is preserved end-to-end *only* for waveform samples that pi-sim chooses to include in the frame; numeric tiles get the latest scalar at SSE rate.

**Fixture and test strategy.** Ship five canonical `current.json` fixtures in the repo: `fixture-stable.json` (healthy vitals, no events), `fixture-alarming.json` (Tachycardia + Hypoxia events active), `fixture-stale.json` (timestamp 5 s in the past), `fixture-offline.json` (intentionally absent file), and `fixture-invalid.json` (NaN in HR field). Drive Playwright tests against each by pointing the Node bridge at the fixture path and freezing `Date.now()` with `page.clock.install()` so the freshness math is deterministic. Required Playwright assertions: numeric tiles update within 500 ms of a frame change; the STALE pill appears at exactly 2 s after the last frame and the OFFLINE banner at 10 s; the alarm row renders the right chips when events are present; no element matching `[data-action="commit"]` or text matching `/save|commit|chart/i` is present anywhere in the monitor module's DOM.

**What not to copy from upstream Pulse UIs.** Skip everything Qt/MOC/VTK from Explorer; skip everything in Unity's `Plugins/`, `Editor/`, and `PulseActionOnClick`; skip all Unreal Blueprint and UMG specifics. Do not pull in any GPL/LGPL components — Pulse is uniformly Apache 2.0 but the **SOFA Pulse plugin** mixes SOFA's LGPL with Pulse, so avoid SOFA derivative code. Do not borrow shader-based waveform glow effects from the game-engine UIs — they will not translate and they trade clarity for prettiness. Above all, do not copy any control that sends actions to Pulse: pi-chart's monitor is read-only by design.

## Final recommendation to a coding agent

**Build:** a React + TypeScript "Live Monitor" module in pi-chart, fed by a Node/Express + chokidar + SSE bridge over `pi-sim/vitals/current.json`, rendering memoized numeric tiles for HR/BP/MAP/RR/SpO₂/Temp plus a sim-time clock, an event-driven alarm row sourced from Pulse's `SEEvent`/`eHeartRhythm` lists, and a four-state freshness indicator (fresh / stale / offline / invalid). Mirror Pulse Explorer's visual hierarchy and color conventions. Use the Unity Asset's `PulseEngineDriver` / `PulseDataLineRenderer` / `PulseDataNumberRenderer` as the architectural pattern for a tick-driven driver plus per-channel renderers. Preserve Pulse's exact property names (`HeartRate`, `SystolicArterialPressure`, `MeanArterialPressure`, `RespirationRate`, `PulseOximetry`, `CoreTemperature`) and units (`mmHg`, `1/min`, `unitless 0–1`, `degC`) on the wire and convert only at render. Persist nothing.

**Do not build:** any control that mutates sim state, any EHR commit affordance, any code path writing to `vitals.jsonl`, audible alarms (defer), multi-patient or historical-trend views (defer), client-side threshold logic (let Pulse's event system be the source of truth for abnormality), or a WASM/Emscripten Pulse build (Kitware doesn't ship one and you don't need one — pi-sim is upstream).

**Risks ranked:** (1) **false-truth UX** — users will treat the monitor as the chart unless labeling, separation, and the absence of commit buttons make the distinction visceral; (2) **drift between display and persistence** if a future adapter writes `vitals.jsonl` from a different timing or rounding policy than the monitor displays — keep both anchored to identical Pulse property names and unit strings; (3) **license contamination** if anyone pulls UI code from the Unity Asset Store package or the Unreal Marketplace download instead of GitLab — those distributions carry storefront EULAs in addition to Apache 2.0; (4) **performance** if waveforms are rendered in React rather than imperatively on Canvas with rAF; (5) **stale-state ambiguity** if freshness is computed off server time instead of `receivedAt`.

**Inspect first, in this order:** `https://pulse.kitware.com/_scenario_file.html` (the canonical Vitals Monitor scenario template — copy its channel set verbatim into pi-sim's frame schema); `https://pulse.kitware.com/_c_d_m_tables.html` (the full `eEvent` and `eHeartRhythm` enumerations — drive your alarm taxonomy from these strings); `https://github.com/agirault/PulseUnity/tree/master/Scripts` for `PulseDataLineRenderer.cs` and `PulseDataNumberRenderer.cs` (clean reference for sweep waveform and tile patterns) and `Scenes/VitalsMonitor.unity` (open in Unity Editor to extract layout dimensions, colors, fonts); `https://pulse.kitware.com/Images/PulseExplorer-Vitals.png` and `https://www.kitware.com/main/wp-content/uploads/2022/03/ExplorerScreen-1024x646.jpg` (visual fidelity reference); the Discourse threads `discourse.kitware.com/t/unity-ecg-data/1027` and the Pulse Pleth thread for the `ArterialPressure → pleth proxy` convention.

**License posture:** Pulse engine, Explorer, Unity Asset (GitLab), Unreal Plugin (GitLab), and Jupyter notebooks are all **Apache 2.0** — verified on `pulse.kitware.com` footer and the GitLab project landing pages, with the Apache 2.0 license file referenced at `https://gitlab.kitware.com/physiology/engine/blob/master/NOTICE`. pi-chart can adapt patterns and copy textual material like enum lists with attribution and a NOTICE file. **Pull source from Kitware GitLab, never from the Unity Asset Store package or the Unreal Marketplace listing**, both of which add storefront EULAs that complicate redistribution. There is no GitHub mirror under `kitware/pulse` despite being a natural URL — the canonical home is `gitlab.kitware.com/physiology/engine`. ## Conclusion

The clearest finding is also the most freeing: **no one has shipped a web-native Pulse vitals monitor**, so pi-chart is not behind a moving target — it is the first. The Apache 2.0 licensing across the suite removes legal friction; the SEEvent-driven alarm pattern removes the hardest UX-correctness question (you don't compute thresholds, the engine does); and the `SEDataRequest` schema gives a stable wire contract between pi-sim and pi-chart. The two non-obvious traps to internalize are that **Pulse's "SpO₂" is `PulseOximetry` not `OxygenSaturation`** and that **saturations are unitless 0–1 not percent** — getting either wrong by one variable name produces a display that is silently wrong rather than visibly broken. The single most valuable design discipline pi-chart can adopt is to make the monitor module physically and visually incapable of being mistaken for the chart of record: distinct theme, no commit affordances, persistent "Live Monitor" labeling, and a hard separation between the SSE display path and any future `vitals.jsonl` adapter. Build the module so that if the persistence layer disappears tomorrow, the monitor still works — and so that if the monitor crashes, no documentation is lost. That asymmetry, more than any visual fidelity to Pulse Explorer, is what makes a clinical live monitor trustworthy.
